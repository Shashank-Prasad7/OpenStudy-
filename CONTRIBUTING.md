# Contributing

1. Create a focused branch from the latest main branch.
2. Keep frontend, backend, and infrastructure changes separated when practical.
3. Use conventional commits such as `feat:`, `fix:`, `docs:`, `style:`, and `test:`.
4. Run the relevant validation commands before opening a pull request.
5. Never commit `.env` files, API keys, database credentials, or generated build directories.

Pull requests should explain the user-visible change, technical approach, test evidence, and any environment or migration changes.
